既有MySQL資料庫與Django整合

Django內建insepctdb可以從既有資料庫自動生成models.py內容。並可設定同步整合，讓Django可以直接操作資料庫。

事前準備

  1. 建好的資料庫(MySQL為例)

  2. Django app (確定已加入settings.py的INSTALLED_APPS內)

  3. mysqlclient套件 (Django 官方文檔推薦)

正式開始

設定settings.py

根據官方文檔說明，先在Django project底下的settings.py檔案中設定database。

詳細參數可參考官方文檔。

自動生成models.py

inspectdb指令會將自動生成的models.py存到該app目錄底下。而在官方文檔則提到:

This feature is meant as a shortcut, not as definitive model generation. See the** documentation of inspectdb** for more information.

因此建議如果資料庫較複雜、或有外鍵等等的，還是要確認一下相關設定。

在專案根目錄底下執行 python manage.py inspectdb > app_name/models.py

此指令會使models.py被覆蓋，若需確認內容建議存到另外檔案名稱，正確後再手動覆蓋。

app_name指的是app名稱，可更改。

執行後需確定models.py編碼是UTF-8，我執行時編碼會是UTF-16，目前還不清楚原因QQ。若編碼非UTF-8，請務必調整。否則後續遷移會出錯。

設定是否讓Django管理資料表

官方文檔內提到:

By default, inspectdb creates unmanaged models. That is, managed = False in the model’s Meta class tells Django not to manage each table’s creation, modification, and deletion:

這裡指的table’s creation, modification, and deletion，就是資料表的管理，即使設定為False還是能透過ORM或SQL指令修改資料數據，因此若不熟悉操作可先維持False。

開啟此功能則當models.py修改時，資料表也會變動。

遷移(Migration)

當設定讓Django管理資料表後，models.py有變動時須執行

python manage.py makemigrations

此時會自動新增一個migrations資料夾在app目錄內，並且會創建紀錄變動的.py檔。

若要讓變更遷移到資料庫，則要輸入

python manager.py migrate

而migrate除了遷移以外，還會創建在INSTALLED_APPS裡預設的app相關資料表。

未來若有models.py更動，都需要下這兩個指令，才能確保遷移成功。
