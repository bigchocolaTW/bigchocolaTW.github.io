組合式api的ref到底是什麼?

之前在前公司使用vue時，使用的是選項式api。並且沒有很好的理解vue這個框架到底做了什麼，那時候就是無腦v-model綁資料而已。

也因此在這次重新好好學習vue3時出現了許多問題，響應式到底是甚麼?ref是什麼?為什麼在組合式api要用ref?

網頁到底有什麼?

  • 介面 UI

  • 事件 event

  • 資料 data

當你按下一個【選我】的按鈕後，畫面上出現了【小西瓜】三個字。

這時候【選我】，是一個**介面(UI)，而你點擊，是一個事件(event)，小西瓜這筆資料(data)則被叫到介面(UI)**上。

若用javascript操作資料顯示時，需要大量繁複的操作dom，這也是為什麼要使用前端框架。不需要繁瑣的操作這些呈現、減少冗長的開發過程，讓框架在背後為我們處理這些事情。沒錯，事情沒有消失，而是框架幫我們處理了。

我想未來我在學習一套框架之前，第一步一定是先了解，他為我做了什麼，而不是我該如何使用它。

vue為我做了什麼?如官網所宣傳的，是一個漸進式框架，並且完全響應式的渲染。

漸進式代表你可以不用一次使用全部的功能，而是逐步依據專案需求採用。狀態管理、路由、你可以再有需求的時後再安裝引入，而不需要一開始就全部裝入你的專案。

響應式則是vue在背後為你監聽、數據綁定、計算屬性等等複雜的事情，這些資料再官方文件以及網路上都有相當人做深入的說明。但目前我想先釐清的就是，那ref跟我原本的選項式api差異到底是什麼?

使用選項式api時，因為會在data選項中定義所有要用到的變數、物件、陣列。

而vue會讓這些資料自動變成響應式因此不需要使用ref。他背後其實為這些資料都被帶到this上。因此叫用這些資料時，都要加上this。

那組合式api呢?使用組合式api時，因為資料沒有明確的區塊分界，而是使用者依照自己的風格，將程式碼依功能分塊，此時才具有響應性。若不使用ref，在組合式api中，是無法直接使用v-model完成雙向綁定的，因為vue不知道他們需要響應式。

而這背後不是透過this，是ref背後還有他的機制，此處不做說明。也因此使用變數時不需加上this。

ref要更改資料內容需要加上.value。

可參考:【Vue 3】ref 跟 reactive 我該怎麼選!? —Mike

雖然可能會覺得，這樣不是比較麻煩嗎?但其實，這麼做相對靈活，因為不是所有東西都需要有響應式，這反而減少了負擔，讓網頁性能提升，也能更清楚看出哪些東西是需要響應式的，對後期開發維護也是一個幫助。
